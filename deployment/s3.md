# AWS S3 storage

Kerf uses S3 as its object-storage backend on AWS. S3 is what Kerf's
`STORAGE_BACKEND=s3` path was written against first — no adapter needed,
no endpoint URL override. The bucket stores project blobs, mesh
tessellations, thumbnails, and Workshop content.

## Why S3 (on AWS deployments)

- **Native S3 API** — no interop shim, full multipart support, no
  credential translation layer.
- **IAM role-based auth** — ECS task role grants `s3:PutObject` /
  `s3:GetObject` directly; no long-lived access keys required.
- **af-south-1** — S3 is available in Cape Town. Data stays in SA;
  no cross-region egress from your ECS tasks.
- **Lifecycle + Glacier** — mature tiering and archival support for
  cold blobs.

If you are running on GCP, Koyeb, or Fly (CPU-only self-host) and want
to use S3 cross-cloud, this works — you will pay $0.09/GB internet
egress from those platforms to S3. For same-cloud storage, use
[gcs.md](./gcs.md) (GCP) or [tigris.md](./tigris.md) (Tigris is
accessible from any host; see that guide for details).

## Create the bucket

```sh
REGION=af-south-1
BUCKET=kerf-blobs-prod

aws s3api create-bucket \
  --bucket "${BUCKET}" \
  --region "${REGION}" \
  --create-bucket-configuration LocationConstraint="${REGION}"
```

For `us-east-1`, omit `--create-bucket-configuration` (it's the default).

### Block public access (required)

```sh
aws s3api put-public-access-block \
  --bucket "${BUCKET}" \
  --public-access-block-configuration \
    "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"
```

### Enable server-side encryption

```sh
aws s3api put-bucket-encryption \
  --bucket "${BUCKET}" \
  --server-side-encryption-configuration '{
    "Rules": [{
      "ApplyServerSideEncryptionByDefault": {
        "SSEAlgorithm": "AES256"
      },
      "BucketKeyEnabled": true
    }]
  }'
```

For KMS encryption (compliance requirements), replace `AES256` with
`aws:kms` and add `KMSMasterKeyID` pointing to your KMS key ARN.

## IAM policy for ECS task role

The ECS task role grants the running container access to the bucket
without long-lived keys:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::kerf-blobs-prod/*"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::kerf-blobs-prod"
    }
  ]
}
```

```sh
aws iam put-role-policy \
  --role-name kerf-ecs-task \
  --policy-name kerf-s3-access \
  --policy-document file://s3-policy.json
```

When the task role is configured, Kerf does **not** need
`KERF_STORAGE_S3_ACCESS_KEY` or `KERF_STORAGE_S3_SECRET_KEY` — the SDK
picks up credentials from the EC2/ECS metadata endpoint automatically.
Set only:

```sh
STORAGE_BACKEND=s3
KERF_STORAGE_S3_BUCKET=kerf-blobs-prod
KERF_STORAGE_S3_REGION=af-south-1
```

If you are running outside AWS (local dev, CI), you will need explicit
access keys or an assumed-role profile.

## Map env vars

| Value | Kerf env var |
|---|---|
| `kerf-blobs-prod` | `KERF_STORAGE_S3_BUCKET` |
| `af-south-1` | `KERF_STORAGE_S3_REGION` |
| (not set — uses task role) | `KERF_STORAGE_S3_ACCESS_KEY` |
| (not set — uses task role) | `KERF_STORAGE_S3_SECRET_KEY` |
| (not set — uses default SDK endpoint) | `KERF_STORAGE_S3_ENDPOINT` |
| `s3` | `STORAGE_BACKEND` |

## Versioning (recommended)

Enable versioning so accidental deletes or overwrites are recoverable:

```sh
aws s3api put-bucket-versioning \
  --bucket "${BUCKET}" \
  --versioning-configuration Status=Enabled
```

List all versions of an object:

```sh
aws s3api list-object-versions --bucket "${BUCKET}" --prefix "projects/user-123/"
```

Restore a previous version by copying it:

```sh
aws s3api copy-object \
  --copy-source "${BUCKET}/projects/user-123/model.step?versionId=xxxxxx" \
  --bucket "${BUCKET}" \
  --key "projects/user-123/model.step"
```

## Lifecycle policies

Delete noncurrent versions to control storage costs. Also transition
old blobs to cheaper storage classes:

```json
{
  "Rules": [
    {
      "ID": "delete-old-noncurrent",
      "Status": "Enabled",
      "Filter": {},
      "NoncurrentVersionExpiration": {"NoncurrentDays": 30}
    },
    {
      "ID": "glacier-old-blobs",
      "Status": "Enabled",
      "Filter": {"Prefix": "blobs/"},
      "Transitions": [
        {"Days": 90, "StorageClass": "STANDARD_IA"},
        {"Days": 365, "StorageClass": "GLACIER_IR"}
      ]
    },
    {
      "ID": "expire-tessellations",
      "Status": "Enabled",
      "Filter": {"Prefix": "tessellations/"},
      "Expiration": {"Days": 30}
    }
  ]
}
```

```sh
aws s3api put-bucket-lifecycle-configuration \
  --bucket "${BUCKET}" \
  --lifecycle-configuration file://lifecycle.json
```

Storage class costs (af-south-1, verify on AWS pricing):
- Standard: $0.023/GB-mo
- Standard-IA: $0.0125/GB-mo (30-day minimum, retrieval charge)
- Glacier IR: $0.004/GB-mo (instant retrieval, higher retrieval cost)

## CORS configuration

If the frontend uploads directly to S3 via presigned PUT URLs, add CORS:

```json
[
  {
    "AllowedOrigins": ["https://kerf.example.com"],
    "AllowedMethods": ["PUT", "POST", "GET", "HEAD"],
    "AllowedHeaders": ["Content-Type", "Content-MD5", "Authorization"],
    "MaxAgeSeconds": 3600
  }
]
```

```sh
aws s3api put-bucket-cors \
  --bucket "${BUCKET}" \
  --cors-configuration file://cors.json
```

## Limits worth knowing

- **Per-object size**: 5 TiB.
- **Multipart upload**: required for objects > 5 GB; recommended above
  100 MB. Kerf's chunked-upload helpers handle this automatically for
  large STEP files and meshes.
- **Incomplete multipart uploads**: add a lifecycle rule to abort
  incomplete multipart uploads after 7 days to avoid paying for partial
  uploads:

  ```json
  {"Rules": [{"ID": "abort-mpu","Status": "Enabled","Filter": {},
    "AbortIncompleteMultipartUpload": {"DaysAfterInitiation": 7}}]}
  ```

## Local dev

For local dev, use MinIO as a drop-in:

```sh
STORAGE_BACKEND=s3
KERF_STORAGE_S3_ENDPOINT=http://localhost:9000
KERF_STORAGE_S3_BUCKET=kerf-local
KERF_STORAGE_S3_ACCESS_KEY=minioadmin
KERF_STORAGE_S3_SECRET_KEY=minioadmin
KERF_STORAGE_S3_REGION=us-east-1
```

The `docker-compose.yml` includes a MinIO service for this.

To test against the real AWS bucket from your local machine, configure
an AWS profile and leave `KERF_STORAGE_S3_ENDPOINT` unset:

```sh
export AWS_PROFILE=kerf-dev
export KERF_STORAGE_S3_BUCKET=kerf-blobs-dev
export KERF_STORAGE_S3_REGION=af-south-1
```

## Troubleshooting

- **403 AccessDenied on PutObject**: the ECS task role is likely missing
  the `s3:PutObject` permission, or the bucket name in the policy doesn't
  match. Double-check the policy ARN and bucket name.
- **NoSuchBucket**: bucket names are global and must match exactly. The
  error also appears if the bucket is in a different region from what
  `KERF_STORAGE_S3_REGION` is set to.
- **SignatureDoesNotMatch (outside ECS)**: if using explicit access keys,
  check the key is active and `KERF_STORAGE_S3_REGION` matches the
  bucket's actual region.
- **Multipart upload leaves partial data**: add the abort-incomplete-MPU
  lifecycle rule above.
- **Egress charges building up**: S3 to ECS in the same region is free.
  If you see egress, something is routing through the internet (e.g., a
  NAT gateway without an S3 VPC endpoint). Add an S3 Gateway VPC
  Endpoint to your VPC:
  ```sh
  aws ec2 create-vpc-endpoint \
    --vpc-id vpc-xxxx \
    --service-name com.amazonaws.af-south-1.s3 \
    --route-table-ids rtb-xxxx
  ```
  S3 VPC endpoints are free and eliminate NAT gateway egress charges.
